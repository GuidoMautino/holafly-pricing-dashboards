Validación del análisis B2B (plan por día)

Antes de avanzar quiero que Data corrobore los supuestos y el dato. No es aprobación de pricing (eso lo ve Pricing/B2B) — es confirmar que las tablas, campos y supuestos están bien. Mapeé el modelo con dos agentes (Gemini) y verifiqué todo contra el dato con queries (varias cosas de los agentes salieron mal y las corregí). Igual prefiero que me corrijan donde haga falta.

Para contexto: el análisis concluyó que un €3/día plano no llega al 40% de margen, pero tieriado por destino sí (~69% mezclado). Eso es lo que va a Pricing. Acá te pido validar la base de dato sobre la que se apoya, no la decisión de precio.

03Lo que asumí — confirmá o corregí

  • 1Aislar B2B por PRODUCTO, no por línea de negocioEl mismo producto aparece etiquetado en business_line distintos (b2b/subscription/b2c); filtrar por business_line='b2b' perdía ~35% de las órdenes. ¿Es correcto aislar por producto? ¿Hay otros SKUs B2B a sumar?
  • 2Ingreso = net_revenue_eur con is_revenue_confirmed=true¿Es el campo y el filtro correctos para el ingreso neto B2B?
  • 3Costo = total_cost_eur por país de consumoLo tomé como euros reales, linkeado por order_id. ¿Correcto? ¿country_code_iso3 ahí es el destino de consumo (no el comprador)?
  • 4Margen = (ingreso − suma de costo en todos los países) ÷ ingreso, por ordenEl plan es mundial, así que el costo de una orden suma el consumo de todos los países donde roameó. ¿Es la forma correcta?
  • 5Ventana anclada en la orden, consumo completoFiltré por created_at y traje TODO el consumo de esas órdenes, sin cortar el consumo por fecha (para no truncar costo). ¿OK?
  • 6Base de "días" (del que menos seguro estoy)orders_per_esim.days vino como centinela -1 (no es duración) y valid_to vacío — el unlimited es una suscripción abierta, sin "días de viaje". Para simular un precio por día usé los días con consumo real como base conservadora. ¿Es razonable? ¿Cuál es la base de días/duración correcta para el unlimited?
  • 7total_cost_eur está en euros realesUn export externo daba valores ~100x más altos (parecía otra unidad). ¿Confirmás que total_cost_eur es el euro real?

04Las queries (para que las corras si querés)

Estas son las que sostienen los supuestos de arriba — cada una dice qué respalda. De las ~15 que corrí, estas 6 son las que importan para corroborar; el resto fueron iteración y debugging (variantes, barridos de precio). Si querés las demás, te las paso.

Universo de productos B2B · respalda el supuesto 1
SELECT ol.product_id,
  COUNT(DISTINCT ol.order_id) AS ordenes_total,
  COUNT(DISTINCT IF(DATE(o.created_at) BETWEEN '2026-02-01' AND '2026-06-04', ol.order_id, NULL)) AS ordenes_ventana
FROM `hly--data--pro--57`.mdm_gold.dim_order_line ol
JOIN `hly--data--pro--57`.mdm_gold.dim_order o ON ol.order_id = o.order_id
WHERE ol.product_id LIKE '%b2b%'
GROUP BY ol.product_id ORDER BY ordenes_ventana DESC;
business_line repartido · respalda el supuesto 1 (por qué NO usamos business_line)
WITH b2b AS (
  SELECT DISTINCT order_id FROM `hly--data--pro--57`.mdm_gold.dim_order_line
  WHERE product_id = 'plan-unlimited-b2b'
)
SELECT f.business_line, COUNT(DISTINCT f.order_id) AS ordenes, ROUND(SUM(f.net_revenue_eur),0) AS net_eur
FROM `hly--data--pro--57`.mdm_gold.fct_revenue f JOIN b2b ON f.order_id = b2b.order_id
GROUP BY f.business_line ORDER BY ordenes DESC;
-- resultado: el mismo producto cae en b2b / subscription / b2c (por eso no se filtra por business_line)
Órdenes sin consumo: legacy vs core · confirma que todo el cohorte es core
WITH b2b AS (
  SELECT DISTINCT ol.order_id, o.platform
  FROM `hly--data--pro--57`.mdm_gold.dim_order_line ol
  JOIN `hly--data--pro--57`.mdm_gold.dim_order o ON ol.order_id = o.order_id
  WHERE ol.product_id LIKE '%b2b%' AND DATE(o.created_at) BETWEEN '2026-02-01' AND '2026-06-04'
),
cost AS (SELECT DISTINCT order_id FROM `hly--data--pro--57`.dp_consumption.consumption_cost_per_day)
SELECT b.platform,
  CASE WHEN c.order_id IS NULL THEN 'sin_consumo' ELSE 'con_consumo' END AS flag,
  COUNT(*) AS ordenes
FROM b2b b LEFT JOIN cost c ON b.order_id = c.order_id
GROUP BY b.platform, flag ORDER BY b.platform, flag;
A · Costo por día de uso, por destino (insumo del tiering)
-- Solo plan-unlimited-b2b. Excluye el eSIM roto de Nepal.
WITH b2b AS (
  SELECT DISTINCT ol.order_id
  FROM `hly--data--pro--57`.mdm_gold.dim_order_line ol
  JOIN `hly--data--pro--57`.mdm_gold.dim_order o ON ol.order_id = o.order_id
  WHERE ol.product_id = 'plan-unlimited-b2b'
    AND DATE(o.created_at) BETWEEN '2026-02-01' AND '2026-06-04'
)
SELECT
  c.country_code_iso3,
  COUNT(DISTINCT c.iccid) AS esims,
  COUNT(DISTINCT CONCAT(c.iccid, CAST(c.utc_date AS STRING))) AS esim_dias,
  ROUND(SUM(c.total_cost_eur), 2) AS cost_eur,
  ROUND(SUM(c.total_cost_eur) / NULLIF(COUNT(DISTINCT CONCAT(c.iccid, CAST(c.utc_date AS STRING))),0), 2) AS cost_per_active_day
FROM `hly--data--pro--57`.dp_consumption.consumption_cost_per_day c
JOIN b2b ON c.order_id = b2b.order_id
WHERE c.iccid != '8943108150106321349'  -- eSIM roto de Nepal
GROUP BY c.country_code_iso3 ORDER BY esim_dias DESC;
-- resultado: 87% del volumen con costo/día <= €1,80; rango €0,18 (UK) a €4+ (caros)
B · Margen mezclado a precio actual + cola negativa
WITH b2b AS (
  SELECT DISTINCT ol.order_id
  FROM `hly--data--pro--57`.mdm_gold.dim_order_line ol
  JOIN `hly--data--pro--57`.mdm_gold.dim_order o ON ol.order_id = o.order_id
  WHERE ol.product_id LIKE '%b2b%'
    AND DATE(o.created_at) BETWEEN '2026-02-01' AND '2026-06-04'
),
rev AS (
  SELECT order_id, SUM(net_revenue_eur) AS net_rev
  FROM `hly--data--pro--57`.mdm_gold.fct_revenue
  WHERE order_id IN (SELECT order_id FROM b2b) AND is_revenue_confirmed = true
  GROUP BY order_id
),
cost AS (
  SELECT order_id, SUM(total_cost_eur) AS cost
  FROM `hly--data--pro--57`.dp_consumption.consumption_cost_per_day
  WHERE order_id IN (SELECT order_id FROM b2b)
  GROUP BY order_id
)
SELECT
  COUNT(*) AS planes,
  ROUND(SUM(r.net_rev),0) AS rev_total,
  ROUND(SUM(COALESCE(c.cost,0)),0) AS cost_total,
  ROUND((SUM(r.net_rev)-SUM(COALESCE(c.cost,0)))/NULLIF(SUM(r.net_rev),0),4) AS gm_mezclado,
  COUNTIF((r.net_rev - COALESCE(c.cost,0)) < 0) AS planes_negativos
FROM rev r LEFT JOIN cost c ON r.order_id = c.order_id;
-- resultado: GM ~83% ex-anomalías a precio actual; cola de planes negativos en destinos caros
Diagnóstico del campo days · respalda el supuesto 6 (la base de días)
WITH b2b AS (
  SELECT DISTINCT ol.order_id, ol.product_id
  FROM `hly--data--pro--57`.mdm_gold.dim_order_line ol
  JOIN `hly--data--pro--57`.mdm_gold.dim_order o ON ol.order_id = o.order_id
  WHERE ol.product_id LIKE '%b2b%' AND DATE(o.created_at) BETWEEN '2026-02-01' AND '2026-06-04'
)
SELECT b.product_id, MIN(e.days) AS days_min, MAX(e.days) AS days_max,
  COUNTIF(e.days < 0) AS filas_dias_neg, COUNTIF(e.valid_to IS NULL) AS sin_valid_to
FROM `hly--data--pro--57`.dp_esim.orders_per_esim e JOIN b2b b ON e.order_id = b.order_id
GROUP BY b.product_id;
-- resultado: days = -1 (centinela) y valid_to vacío para unlimited → no hay duración real
